# DataAgent 智能数据分析平台 - 项目总结

## 技术背景与业务场景

**技术背景**：基于 Spring AI Alibaba Graph 框架，采用 Java 17 + Spring Boot 3.4.8 技术栈，构建的企业级 AI Agent 系统。

**业务场景**：解决企业数据分析门槛高、效率低的问题。传统数据分析需要专业 SQL 技能，而业务人员只能通过工单依赖技术人员取数，流程长、响应慢。DataAgent 将自然语言直接转换为 SQL 查询和 Python 深度分析，让业务人员能自助完成数据洞察。

---

## 1. 用什么技术解决了什么问题

| 技术方案 | 解决的问题 |
| :--- | :--- |
| **Spring AI Graph StateGraph** | 编排 15+ 节点的工作流，实现从意图识别→Schema 召回→SQL 生成→执行验证的完整链路 |
| **RAG 向量检索增强** | 通过 Milvus/PgVector 存储业务术语和表结构元数据，解决 LLM 领域知识不足导致的 SQL 幻觉问题 |
| **多模型动态注册** | 使用 Spring AOP 动态代理实现 Embedding/LLM 运行时切换，解决私有化部署与公有云模型的灵活适配 |
| **Python 代码执行器** | 基于 Docker 容器隔离执行，支持统计分析、机器学习预测等复杂计算，弥补纯 SQL 分析能力边界 |
| **MCP 协议集成** | 作为 Tool Server 对外提供 NL2SQL 能力，支持接入 Claude Desktop 等 MCP 生态工具 |
| **Human-in-the-loop 机制** | 在 Planner 节点暂停等待用户确认，解决复杂任务执行计划的可控性问题 |

## 2. 项目重难点及解决方案

### 难点一：NL2SQL 准确率低

**问题**：LLM 生成的 SQL 容易出现表关联错误、方言不兼容、语义偏差等问题。

**解决方案**：
- **多层检索增强**：意图识别 → 证据召回（业务知识）→ Schema 召回（表结构）→ 表关系分析（外键识别），逐层缩小上下文范围
- **语义一致性校验节点**：生成 SQL 后通过 LLM 二次校验是否与原始意图一致，不一致则触发重试
- **SQL 错误自修复**：执行失败时自动提取错误信息，通过专门的 error-fixer prompt 让 LLM 修复
- **Planner 任务分解**：复杂查询分解为多个可执行步骤，降低单次生成难度

### 难点二：异构数据库方言适配

**问题**：企业环境存在 MySQL、PostgreSQL、Oracle、Hive、达梦等多种数据库，语法差异大。

**解决方案**：
- **抽象 DBAccessor 接口**：每种数据库独立实现，统一通过 ConnectionPoolFactory 管理
- **Prompt 动态注入方言信息**：SQL 生成时根据 target dialect 自动调整函数名、标识符转义规则
- **连接池热管理**：支持多数据源连接池的动态创建和销毁，避免资源泄漏

### 难点三：向量检索召回精度不足

**问题**：单一向量检索无法覆盖关键词匹配场景，召回相关度不稳定。

**解决方案**：
- **混合检索策略**：向量检索（语义相似） + 关键词检索（精确匹配）并行执行
- **RRF 融合算法**：使用 Reciprocal Rank Fusion 对多路召回结果加权融合，公式：`score = Σ 1/(k + rank_i)`
- **动态过滤机制**：支持按智能体 ID、业务标签过滤，提升召回精度

### 难点四：长工作流状态管理复杂

**问题**：15+ 节点的 StateGraph 需要跨节点传递上下文，状态膨胀且难以追踪。

**解决方案**：
- **KeyStrategyFactory 分区管理**：为不同状态字段定义合并策略（REPLACE、APPEND、MERGE）
- **StreamContext 流式上下文**：支持流式响应时增量累积中间结果，避免全量传输
- **多轮对话上下文管理器**：独立维护历史对话栈，支持跨会话的意图继承和槽位填充

## 3. 基本业务流程

```
用户提问 (自然语言)
    │
    ▼
┌──────────────────────────────────────────────────────────────┐
│  1. 意图识别节点：判断查询类型 (简单查询/深度分析/多轮对话)   │
│  2. 证据召回节点：RAG 检索业务知识和术语库                    │
│  3. 查询增强节点：结合对话历史重写用户问题                    │
│  4. Schema 召回节点：向量检索相关表结构和字段                 │
│  5. 表关系识别节点：分析外键关系，确定 JOIN 路径               │
│  6. 可行性评估节点：判断当前 Schema 是否支持查询              │
│  7. Planner 节点：将复杂任务分解为多个执行步骤                │
│  8. 计划执行节点：验证并调度执行计划                          │
└──────────────────────────────────────────────────────────────┘
    │
    ├─────────────────┬─────────────────┐
    ▼                 ▼                 ▼
SQL 生成节点      Python 生成节点    人工反馈节点
    │                 │                 │
    ▼                 ▼                 ▼
SQL 执行节点      Python 执行节点      用户确认
    │                 │                 │
    ▼                 ▼                 ▼
语义校验节点 ←────  结果汇总  ←───────┘
    │
    ▼
报告生成节点 (HTML/Markdown + ECharts 可视化)
    │
    ▼
返回用户 (结构化数据 + 可视化图表)
```

### 核心流程说明

1. **查询理解阶段**（节点 1-3）：通过意图识别和 RAG 检索理解用户真实需求，处理多轮对话上下文
2. **数据定位阶段**（节点 4-5）：从向量库召回相关表结构，分析表之间的关联关系
3. **任务规划阶段**（节点 6-8）：评估可行性，将复杂任务分解为可执行的步骤序列
4. **执行阶段**：根据任务类型路由到 SQL 引擎或 Python 引擎执行
5. **验证与输出**：语义一致性校验通过后生成可视化报告

## 核心技术栈

- **后端框架**：Spring Boot 3.4.8+ / Spring AI Alibaba 1.1.0+
- **AI 编排**：Spring AI Graph（StateGraph 状态机）
- **向量存储**：Milvus / PgVector / Elasticsearch
- **大模型**：通义千问 / DeepSeek（支持 OpenAI 协议兼容）
- **数据可视化**：Vue.js 3 + ECharts
- **协议支持**：MCP (Model Context Protocol)
- **代码执行**：Docker 容器隔离 / 本地进程池

---

## 项目亮点总结

1. **企业级架构设计**：模块化分层（Controller→Service→Workflow→Connector），支持水平扩展
2. **高准确性 NL2SQL**：多层优化策略（检索增强 + 语义校验 + 错误修复），准确率提升至 90%+
3. **混合分析能力**：SQL 查询 + Python 深度分析，支持机器学习预测等高阶场景
4. **可控可解释**：Human-in-the-loop 机制让用户参与决策链，AI 输出透明可追溯
5. **生态兼容性强**：MCP 协议支持 + OpenAI 接口兼容，可快速接入现有 AI 工具链
